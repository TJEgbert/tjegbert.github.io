Game Database

A user generated database that as the user inputs the more searchable it becomes.

Why I made it

I like most gamer adults have a huge back log of games stretching between many different platforms and some are bound to get forgotten. That was the main idea going into making a program where I could enter my game collection and have it easily searchable and updates as I put more games into it. That was until I asked my brother for his opinion since we are in the same boat. This was when the idea of using an API came from to help fill out some of the information about the game for you. Just from adding in the API this went from a week project into almost two. Also, I need more projects for my portfolio. You can go straight to the repository by clicking here.

Client Info Screen

This window allows the user to enter in the Client ID and Client Info you get from Twitch in the database. This allows the user to be able to connect to the IGDB API later in the Add Game Window. This was the first hurdle of the project was getting the access token needed to be able to connect to IGDB API. With the Client ID and Secret you need to make a Post to Twitch so they can send back the access token and the expiration date. To achieve this, I ended up using RestEase to make the interface and Post with the information to Twitch. From there I used NewtonSoft.Json to then deserialized the returning .Json. This was one of the first big hurdles of the program was getting this all to work here and the Add Game Window. With no prior knowledge I jumped into the RestEase documentation to figure it out. After reading it all I jumped into the API documentation to see what they were expected to receive. They expected Post with the particle path with ClientID and Secret. This was simple enough to do with RestEase make a simple Post with the path once I knew what I was doing. From there it all went to the Microsoft Access files to store for whenever the program needed it use it later.

Main window

This was the window that gave me the least trouble out of all of them. It's also the window that I had put the most thought into before I started to code since nothing huge was added on to it later. All this window is doing is acting as an interface with database. It displays everything the database has in the Game table, and the search filters are generated from the database itself. My favorite moment when I was coding this window was making it so the user could update the records right from there. I saw you could enable editing on data grid, so I had it do that, but I needed to tell if something was edited. This was where the observable collection worked so well. I ended up having a bool on the game object that would be checked true if it was edited on the data grid. Once the user clicks save it removes the entry and puts it back into the database with the updated information. The thing that shocked me the most was it worked on the first try.

Add Game Window

This was the next window that took awhile and that just for the fact of accessing the API. Again, I used RestEase and NewtonSoft.json to access the IGDB API. This process took a while since it's the same as Twitch where you must access it using Post, but they expect specific information in certain places. The header must contain the Client ID and the Access token. Then the path must contain the endpoint with query of what you're asking for. This took me quite a while to get to work. After beating my head against the problem for while I took the advice that was given in the documentation for API. I ended up downloading Insomnia lets you access the API's and make queries like you would with a regular database. This helped me greatly since one of my biggest errors came from syntax with querying that API. They use the Apicalypse syntax which is a lot like SQL but just different enough that it easy to make a mistake. After getting all that worked out from there it was simple filling up the window interface filled up the results from the API and the local database.

Game List window

This window came from my brother with the idea “Wouldn't it be nice to have a list of what you would like to play next.” He is also a computer science major like me, so we started talking one night about how to make it work in the database. The next morning it was time for me to implement this. I'm still not 100% sold on how I did this, there's another way I thought about where it is all done on the main window. I will end up trying this in future, but I need a break from this. With school starting up in a week I want to try to get as much done on a video game as possible before its back to splitting focus. The major challenge of this window was trying to make it so people couldn't leave before it was saved. Luckily .net has some handy features that help you do just that.

Setting window

Again, since my brother brought up a good point about already having some starter content in the program that users could use would be nice. I agreed with him if that this was going to be a program, I was going to sell it to a mass market. I believe that most people who are going to find that repository on GitHub would read the READ ME and then start using that program. So, to ease my mind a little I decided to make it an option if the user wanted it or not. Hence an earlier than expected Settings window was made so the user can toggle it off and on. Since user preference is stored in the database, I can check if it's turned on or not. Since this does involve some loops to make sure no duplicates are added. It is set up that once everything has been used once meaning it's in the database it will turn of this feature so the program on run more efficiently.
